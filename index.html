Understand tokens first

Tokens are the currency of AI. Before using the calculator, spend 2 minutes here — it makes everything else click.

What is a token?

Think of a token as a word-chunk. Claude doesn't read letter by letter — it reads in small pieces called tokens. About 4 letters = 1 token.

"bank"= 1 token
"banking"≈ 2 tokens
1 sentence≈ 15–20 tokens
1 paragraph≈ 80–100 tokens
1 code file≈ 2,000–4,000 tokens
1 million tokens≈ 750,000 words
Input vs output

Input is what you send to Claude. Output is what Claude writes back. Writing costs 5× more than reading — because generating an answer is harder work.

The formula
Cost = Input Cost + Output Cost
Input Cost  = Tokens ÷ 1,000,000 × $3
Output Cost = Tokens ÷ 1,000,000 × $15
Example — medium story on Sonnet 4.6
(25,000 ÷ 1M × $3) + (8,000 ÷ 1M × $15)
= $0.075 + $0.120 = $0.195

The 6 optimization pillars — explained
1. Match model to task — don't default to Opus everywhere
Up to 60% cost saving
Highest impact

Most token waste comes from using Opus when Sonnet would do the job. Route tasks by complexity — not habit.

Task typeRecommended modelWhy
Architecture decisions, security review, complex debuggingOpus 4.8High reasoning needed
Most Jira story implementation, code generation, refactoringSonnet 4.6Best price/quality balance
Routing, classification, simple validation, format checksHaiku 4.5Fast & cheap
Repeated batch processing, overnight jobsHaiku 4.5 + Batch API50% off everything
Claude Code command: Use claude --model claude-haiku-4-5 for routine tasks and claude --model claude-opus-4-8 only when needed.
2. Keep CLAUDE.md under 200 lines — it's a tax on every request
Up to 30% cost saving
High impact

Your CLAUDE.md (or system prompt) is injected into every single API call. Every line you add is a permanent token tax on every turn of every session — forever.

Keep in CLAUDE.md
Project name, tech stack, key conventions
Coding style rules (max 10–15 lines)
Testing approach and commands
File structure overview (brief)
Critical do-nots for your codebase
Remove from CLAUDE.md
Long explanations of how tools work
Full API documentation or specs
Complete list of every file in the repo
Historical context or old decisions
Anything Claude can infer from the code
Target: Keep CLAUDE.md under 200 lines. Every line over that costs you money on every single API call.
3. Add an ignore file — this is the single highest-leverage change
Up to 40% cost saving
Highest impact

Without an ignore file, Claude reads node_modules, build output, binary files, and large data files — thousands of tokens of noise before any actual work. This often outweighs your own prompts as the biggest token drain.

.claudeignore — recommended starting point

# Dependencies (massive — always ignore)
node_modules/
vendor/
.pnp/

# Build output
dist/
build/
.next/
out/
coverage/

# Large data files

*.json.gz
*.sql
*.dump
*.log

# Binary and media
*.png *.jpg *.gif *.ico *.svg
*.pdf *.zip *.tar.gz
*.woff *.woff2 *.ttf

# IDE and OS

.vscode/settings.json
*.pyc
__pycache__/
Create a .claudeignore file in your project root. Claude Code respects it the same way Git respects .gitignore.
4. Clear and compact context between tasks
Up to 35% cost saving
High impact

A fresh session has a small payload per turn. But a long session resends everything before it on every subsequent message — cost grows geometrically with turn count. Stale context is wasted tokens on every message.

/clear
Start a completely fresh context. Use when switching to an unrelated task. Rename the session first so you can find it later.
/compact
Summarizes conversation history when approaching context limits. Preserves key decisions and code changes, drops exploratory back-and-forth.
/usage
Check current token usage at any point in the session. Configure a status line to display it continuously so you spot token-heavy phases early.
Start session → implement story A → /compact or /clear
New session → implement story B → /compact or /clear
One marathon session for story A + B + C + D = token cost grows geometrically
5. Plan before you implement — catch issues before they cost tokens
Up to 25% cost saving
Medium impact

Entering plan mode before complex implementation lets Claude outline an approach before writing code — catching issues early and preventing costly rework loops. A targeted prompt is always better than a huge context window.

claude --plan
Claude outlines its approach first. You review and approve before it writes a single line of code. Prevents expensive rework from misunderstood requirements.
Targeted prompts (efficient)
"Read src/upload/handler.js and its closest tests, implement validation"
"Look at only the document service layer, not the loan flow"
"Fix the bug in line 47 of auth.js — don't read other files"
Vague prompts (wasteful)
"Read all the files and implement the feature"
"Scan the whole codebase and understand it"
"Fix everything that's wrong"
6. Disconnect MCP servers you're not actively using
Up to 20% cost saving
Medium impact

Each connected MCP server loads its full tool definitions into every message. Multiple servers compound — a handful of connected servers can add significant token overhead before any actual work happens.

Wasteful setup
GitHub MCP Slack MCP Jira MCP Google Drive MCP Salesforce MCP
All connected all the time — tool definitions loaded on every message even when not needed
Optimised setup
GitHub MCP Slack MCP (off) Jira MCP Drive MCP (off)
Only connect what you need for this specific task — disconnect the rest

The honest answer — do we have to choose?
Most professional teams use both. Copilot handles daily typing speed in the IDE. Claude Code handles the big tasks — building a full Jira story, large refactors, implementing a whole feature. They work at completely different levels and don't conflict.
